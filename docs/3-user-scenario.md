### 시나리오 간 연관 관계

```
[비인증 사용자]
      │
      ▼
 ┌─────────────┐        실패 시
 │  US-01 온보딩 │──────────────────► US-06 예외/에러
 │  회원가입    │                     (중복 이메일, 유효성 오류)
 └──────┬──────┘
        │ 가입 완료
        ▼
 ┌─────────────┐        실패 시
 │  UC-02 로그인 │──────────────────► US-06 예외/에러
 │  (인증 진입)  │                     (로그인 실패, 비인증 접근)
 └──────┬──────┘
        │ 로그인 성공 (theme/locale 자동 복원)
        │
        ├──────────────────────────────────────────┐
        ▼                                          ▼
 ┌─────────────────┐                    ┌──────────────────┐
 │  US-02 핵심 업무 │                    │  US-05 설정 변경  │
 │  카테고리 생성   │                    │  테마 / 언어 변경  │
 │  할일 CRUD      │                    └────────┬─────────┘
 └──────┬──────────┘                             │ 저장 후
        │ 할일 누적                               │ 로그아웃 → 재로그인
        ▼                                        │ → 설정 자동 복원
 ┌─────────────────┐
 │  US-03 필터링   │
 │  카테고리+상태  │
 │  복합 필터      │
 └──────┬──────────┘
        │ OVERDUE 발견
        ▼
 ┌──────────────────┐       실패 시
 │  US-04 기간 초과  │──────────────────► US-06 예외/에러
 │  완료 처리        │                     (날짜 유효성, 권한 오류)
 │  또는 날짜 수정   │
 └──────────────────┘
```

### 할일 생명주기와 시나리오 매핑

```
         할일 등록 (US-01, US-02)
               │
               ▼
        ┌─────────────┐
        │ NOT_STARTED │ ◄── start_date 미도래 또는 날짜 미설정
        └──────┬──────┘
               │ 시작일 도래
               ▼
        ┌─────────────┐
        │ IN_PROGRESS │ ◄── start_date <= today <= end_date
        └──────┬──────┘
          │         │
          │ 완료처리  │ 종료일 초과
          │ (US-02)  │
          ▼         ▼
       ┌──────┐  ┌─────────┐
       │ DONE │  │ OVERDUE │ ◄── 기간 초과 처리 (US-04)
       └──────┘  └────┬────┘
                      │ 완료처리 또는 날짜 수정
                      ▼
                ┌───────────────┐
                │ DONE 또는     │
                │ IN_PROGRESS   │
                └───────────────┘
```

> 필터링(US-03)은 모든 상태에서 진입 가능하며, 예외/에러(US-06)는 모든 시나리오의 실패 경로에서 파생된다.

## US-01: 온보딩 — 신규 가입 후 첫 할일 등록

**페르소나**: 이지현, 대학교 2학년 (21세). 학기 중 과제와 시험 일정을 체계적으로 관리하고 싶어 Todo List 앱을 처음 접한다.

**목표**: 신규 계정을 생성하고 첫 번째 할일(중간고사 준비)을 등록한다.

**사전 조건**: 앱에 계정이 없는 상태. 브라우저로 서비스 URL에 최초 접근.

### 정상 흐름 (Happy Path)

| 단계 | 사용자 행동 | 시스템 반응 | 관련 UC/BR |
|------|------------|------------|-----------|
| 1 | `/login` 접근 | S-01 로그인 화면 렌더링. 브라우저 언어(ko) 및 시스템 테마 자동 적용 | BR-09 |
| 2 | "회원가입" 링크 클릭 | S-02 회원가입 화면(`/register`)으로 이동 | UC-01 |
| 3 | 이메일(`jh@univ.ac.kr`), 비밀번호(`Study2026`), 이름(`이지현`) 입력 후 "가입하기" 클릭 | 입력값 유효성 검증 통과 → 계정 생성 → '기본' 카테고리 자동 생성 → JWT 발급 → S-03 대시보드(`/`)로 리다이렉트 | UC-01, BR-03 |
| 4 | S-03 대시보드 첫 진입 | 카테고리 사이드바에 '기본' 카테고리 표시. 할일 목록 비어 있음 | UC-06, BR-02 |
| 5 | "+ 할일 등록" 버튼 클릭 | S-04 할일 등록 화면(`/todos/new`)으로 이동 | UC-10 |
| 6 | 제목(`중간고사 운영체제 준비`), 설명(`교재 6~10장 복습`), start_date(`2026-06-01`), end_date(`2026-06-14`) 입력. 카테고리는 선택하지 않음 | — | BR-03, BR-05 |
| 7 | "저장" 클릭 | category_id = '기본' 자동 적용 → 할일 생성 → S-03 대시보드로 이동 → 목록에 신규 할일 표시. status = `NOT_STARTED` (today < start_date) | UC-10, BR-03, BR-06 |

## US-02: 핵심 업무 흐름 — 카테고리 생성부터 할일 관리까지

**페르소나**: 박민준, 직장인 34세. 마케팅팀 PM으로 업무를 프로젝트별로 분류하여 관리한다. 매일 아침 할일을 점검하고 업무 진행 상황을 확인한다.

**목표**: '마케팅' 카테고리를 만들고 오늘 처리할 업무 할일을 등록한다. 이후 진행 중인 할일을 완료 처리한다.

**사전 조건**: 로그인된 상태. 기존에 '기본' 카테고리만 존재. 할일이 2개 있음.

### 정상 흐름 (Happy Path)

| 단계 | 사용자 행동 | 시스템 반응 | 관련 UC/BR |
|------|------------|------------|-----------|
| 1 | S-03 대시보드에서 "카테고리 관리" 메뉴 클릭 | S-07 카테고리 관리 화면(`/categories`)으로 이동. '기본' 카테고리에는 수정/삭제 버튼 비활성화 표시 | UC-06, BR-04 |
| 2 | 카테고리명 입력란에 `마케팅` 입력 후 "추가" 클릭 | 유효성 검증 통과 → 카테고리 생성 → 목록에 '마케팅' 카테고리 즉시 반영 | UC-07, BR-01, BR-02 |
| 3 | 대시보드(`/`)로 돌아감 | 사이드바에 '기본', '마케팅' 카테고리 표시 | UC-06 |
| 4 | "+ 할일 등록" 클릭 → S-04 이동 | 할일 등록 폼 렌더링. 카테고리 셀렉터에 '기본', '마케팅' 선택 가능 | UC-10 |
| 5 | 제목(`2분기 캠페인 기획안 작성`), 카테고리(`마케팅`), start_date(`2026-05-27`), end_date(`2026-05-30`) 입력 후 "저장" | 할일 생성. today = start_date이므로 status = `IN_PROGRESS`. 대시보드 목록에 표시 | UC-10, BR-05 |
| 6 | 목록에서 기존 할일 `보고서 초안 검토` 클릭 | S-06 할일 상세 화면(`/todos/:id`)으로 이동. status 배지, 완료 처리 버튼, 수정/삭제 버튼 표시 | UC-12 |
| 7 | "완료 처리" 버튼 클릭 | PATCH 요청 → is_done = true → status = `DONE`으로 즉시 변경 → status 배지 갱신 | UC-14, BR-01, BR-02 |
| 8 | "수정" 버튼 클릭 | S-05 수정 화면(`/todos/:id/edit`)으로 이동. 기존 값 프리필 | UC-13 |
| 9 | description 내용 수정 후 "저장" | 유효성 검증 통과 → 업데이트 → S-06 상세 화면으로 복귀. 변경 내용 반영 | UC-13, BR-05 |

## US-03: 필터링 — 원하는 할일 목록 탐색

**페르소나**: 박민준, 직장인 34세. 여러 카테고리에 할일이 쌓인 상황에서 '마케팅' 카테고리의 진행 중인 항목만 골라서 확인하려 한다.

**목표**: 카테고리 + 상태 복합 필터로 원하는 할일 목록만 조회한다.

**사전 조건**: 로그인된 상태. 할일 10여 개가 여러 카테고리에 분산되어 있음. 상태 혼재(NOT_STARTED, IN_PROGRESS, DONE, OVERDUE).

### 정상 흐름 (Happy Path)

| 단계 | 사용자 행동 | 시스템 반응 | 관련 UC/BR |
|------|------------|------------|-----------|
| 1 | S-03 대시보드 진입 | 전체 할일 목록 표시 (상태 파생값 포함). 상태 필터 탭(전체/시작전/진행중/완료/기간초과) 표시 | UC-11, BR-02 |
| 2 | 사이드바에서 '마케팅' 카테고리 클릭 | `category_id` 필터 적용 → '마케팅' 카테고리 소속 할일만 표시 | UC-11 |
| 3 | 상태 필터 탭에서 "진행중" 클릭 | `status=IN_PROGRESS` 필터 추가 → category_id AND status 복합 필터 적용 → 마케팅 + 진행중 할일만 표시 | UC-11 |
| 4 | 특정 할일 클릭 | S-06 상세 화면으로 이동. 전체 정보 및 `IN_PROGRESS` status 배지 확인 | UC-12 |
| 5 | 브라우저 뒤로가기 | S-03 복귀. 이전 필터 상태 유지 (마케팅 + 진행중) | — |
| 6 | 사이드바에서 "전체" 클릭, 상태 탭 "전체" 클릭 | 필터 해제 → 전체 할일 목록 복원 | UC-11 |
| 7 | 상태 탭 "기간초과" 클릭 | `status=OVERDUE` 필터 적용 → 기간 초과 할일만 표시 | UC-11 |

### 예외 흐름

| 단계 | 예외 상황 | 시스템 반응 |
|------|----------|------------|
| 3-A | 선택한 조합에 해당하는 할일 없음 | "해당 조건의 할일이 없습니다." 빈 상태 메시지 표시 |

**사후 조건**: 사용자가 원하는 조건의 할일 목록을 성공적으로 조회. 필터 조작 후 데이터 무결성 유지됨.

## US-04: 기간 초과 처리 — 마감 놓친 할일 발견과 처리

**페르소나**: 이지현, 대학교 2학년 (21세). 며칠 전 등록한 과제 제출 할일의 마감일을 놓쳤다는 것을 오늘 아침 앱에 접속해 알게 되었다.

**목표**: OVERDUE 상태의 할일을 발견하고, 수정 또는 완료 처리로 정리한다.

**사전 조건**: 로그인된 상태. `end_date = 2026-05-25`인 할일(`자료구조 과제 제출`)이 미완료 상태로 존재. 오늘은 `2026-05-27`.

### 정상 흐름 (Happy Path)

| 단계 | 사용자 행동 | 시스템 반응 | 관련 UC/BR |
|------|------------|------------|-----------|
| 1 | S-03 대시보드 진입 | 서버가 조회 시점 기준으로 status 재계산. `자료구조 과제 제출`의 status = `OVERDUE` (today > end_date, is_done=false). 목록에 OVERDUE 배지 표시 | UC-11, BR-02 |
| 2 | 상태 탭 "기간초과" 클릭 | OVERDUE 할일만 필터링하여 표시. `자료구조 과제 제출` 확인 | UC-11 |
| 3 | 해당 할일 클릭 | S-06 상세 화면 이동. `OVERDUE` status 배지, 완료 처리 버튼, 수정 버튼 표시 | UC-12 |
| 4-A | [케이스 A] 실제로 과제를 제출했음 → "완료 처리" 클릭 | is_done = true → status = `DONE`으로 즉시 변경. OVERDUE 목록에서 사라짐 | UC-14 |

## US-05: 설정 변경 — 테마·언어 변경 및 재로그인 복원

**페르소나**: 박민준, 직장인 34세. 야근 중 화면이 눈에 부담스러워 다크 모드로 전환하고, 영어 UI로도 변경해 보려 한다. 다음날 다시 로그인해도 설정이 유지되길 원한다.

**목표**: 다크 모드와 영어 언어를 설정하고, 로그아웃 후 재로그인 시 설정이 자동 복원되는 것을 확인한다.

**사전 조건**: 로그인된 상태. 현재 theme = `light`, locale = `ko`.

## US-06: 예외/에러 — 유효성 오류 및 권한 오류 처리

**페르소나**: 복합 시나리오. 유효하지 않은 입력, 권한 없는 접근, 만료된 세션 등 실패 흐름을 다룬다.

**목표**: 주요 에러 상황에서 시스템이 적절한 피드백을 제공하고 사용자가 복구할 수 있음을 검증한다.

**사전 조건**: 상황별로 다름 (각 예외 흐름 참조).

| 단계 | 사용자 행동 | 시스템 반응 | 관련 UC/BR |

| 1 | `/login`에서 존재하지 않는 이메일 또는 틀린 비밀번호 입력 후 로그인 시도 | HTTP 401. "이메일 또는 비밀번호가 올바르지 않습니다." 로그인 폼 유지. JWT 미발급 | UC-02 |
